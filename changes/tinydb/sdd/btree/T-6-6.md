# T-6.6 — Recovery (REDO / UNDO) — B6 gate batch

## Goal
On startup, replay the WAL:
- **REDO** committed transactions: apply the page state changes.
- **UNDO** uncommitted transactions: roll back the page state changes.

This requires that EVERY page write is logged as an `RT_PAGE` record holding the before-image (for UNDO) and after-image (for REDO). The Transaction Manager must wire DML to write PAGE records BEFORE and AFTER each data-page modification.

This is the gate test for Batch 6 — a **fuzzy test** of 100 random operation sequences must pass.

## Carry-forward NITs

- v0.1 page-level REDO/UNDO: simplest model is "before/after image per page write". No fine-grained log records (B9 polish).
- `WAL.append(RT_PAGE, payload)` already exists; the manager will use it.
- The B5 DML paths (`Insert`/`Update`/`Delete` in `src/tinydb/executor/dml.py`) currently write directly to heap pages. They need to be wrapped with PAGE logging.
- Catalog pages and index pages also need logging — pass the `page_writer` callback into the manager so any code that flushes pages emits a `RT_PAGE` record.

### `tests/tx/test_recovery.py` (NEW — the gate)

The gate test: **100 random operation sequences followed by simulated crash → reopen → state must match expected**.

```python
import random
import shutil
from pathlib import Path
import pytest

@pytest.mark.timeout(60)
def test_fuzzy_recovery_100_sequences(tmp_path):
    """The B6 gate: 100 random sequences of BEGIN/INSERT/UPDATE/DELETE/COMMIT
    followed by a simulated crash and recovery. After recovery the database
    must contain EXACTLY the committed state.
    """
    for seed in range(100):
        _scenario(tmp_path, seed)
```

Each scenario:
1. Open a fresh `.db` + `.wal`.
2. Run a sequence of random operations — BEGIN, INSERT (random row), UPDATE (random id), DELETE (random id), COMMIT. Random ROLLBACK.
3. Randomly decide to "crash" mid-sequence: do NOT fsync, do NOT run any commit/rollback for the last open tx.
4. Reopen the database: open WAL → run `Recovery.replay()` → reopen Catalog / Heap → verify rows match the committed state.

This is the test that proves durability + atomicity + consistency across crashes.

## Tests Required

`tests/tx/test_recovery.py`:

1. REDO a committed INSERT → page shows the new row.
2. UNDO an uncommitted INSERT → page is empty.
3. REDO a committed UPDATE → row has new value.
4. UNDO an uncommitted UPDATE → row has old value.
5. REDO a committed DELETE → row is tombstoned.
6. UNDO an uncommitted DELETE → row restored.
7. Two transactions interleaved (T1 BEGIN tx1, T2 BEGIN tx2, T1 COMMIT, T2 CRASH) → only T1's changes survive.
8. Recovery is idempotent — calling replay() twice doesn't double-apply.
9. Empty WAL → recovery is a no-op.
10. WAL with only ROLLBACK records → no-op (no REDO).
11. **Fuzzy test: 100 sequences with simulated crashes** (this is the gate).
