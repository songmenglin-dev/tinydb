# T-6.1 — Write lock

## Goal
Introduce `src/tinydb/tx/` package with a simple **single-writer** write lock (per REQ-TRX-4 "single writer semantics"). The lock is **reentrant** (same thread can re-acquire without deadlock) and **context-managed**. This is the foundation for B6: every later task wraps DML in this lock.

### `src/tinydb/tx/lock.py`
```python
"""Single-writer write lock with reentrance.

Tinydb v0.1 is a single-process, single-connection database (per
spec-superflow DP-0 scope fence — no concurrent writers in
v0.1). The WriteLock enforces "at most one active writer" using an
in-process recursion-depth counter on the owning thread.

API
---
    lock = WriteLock()
    with lock.acquire() as held:
        # critical section; `held` is a WriteLockHeld token
        ...

Reentrance: the same thread may acquire the lock multiple times.
Each acquire() increments the depth; release decrements. Only when
depth returns to zero is the lock released. Other threads block
on a Condition.wait().

Non-reentrance from another thread: blocks until the holder
releases.

This is intentionally tiny. We do NOT need:
- Read/write locks (single-process ⇒ no concurrency)
- Fair scheduling (single-process ⇒ not observed)
- Lock timeouts (out of scope for v0.1; documented)
"""
from __future__ import annotations
import threading
from types import TracebackType
from typing import Optional, Type

class WriteLockHeld:
    """Token returned by WriteLock.acquire() — opaque marker.

    Holds a (Lock, depth) pair; releasing decrements depth.
    """

class WriteLock:
    """Single-writer reentrant lock (one writer at a time).

    Thread-safety: lock state is held in self._owner + self._depth +
    self._cond. Multiple writers block on Condition.wait() until the
    owner releases (depth == 0).
    """

    def __init__(self) -> None:
        self._lock = threading.Lock()  # protects _owner, _depth, _cond
        self._owner: Optional[int] = None  # thread id
        self._depth: int = 0
        self._cond = threading.Condition(self._lock)

    def acquire(self) -> WriteLockHeld:
        """Block until the calling thread owns the lock. Reentrant."""
        ...

    def release(self) -> None:
        """Drop one reentrance level; if depth==0, wake the next waiter."""
        ...

    # context-manager sugar
    def __enter__(self) -> WriteLockHeld: ...
    def __exit__(self, et, ev, tb) -> bool: ...
```

> Use `thread.get_ident()` to identify the holder, not `threading.current_thread()` (faster and avoids reference loops). The condition variable approach is the canonical Python idiom; do NOT roll your own with Event.

## Tests Required

`tests/tx/__init__.py` — empty file.
`tests/tx/conftest.py` — minimal; tests may not need fixtures.
`tests/tx/test_lock.py` — at least 8 cases:

1. `lock.acquire()` then `lock.release()` from same thread → no error.
2. Nested acquire (3 levels) then release 3 times → depth returns to 0.
3. Two different threads: one holds, the other blocks until release.
4. Two threads acquiring simultaneously: only one wins; second proceeds after release.
5. Reentrant: thread A acquires twice, thread B starts → blocks.
6. After thread A depth returns to 0, thread B unblocks and proceeds.
7. Context manager: `with lock.acquire() as held: ...` works.
8. Reentrance via context manager: nested `with` blocks increase depth.
9. (bonus) Many threads contending — only one runs at a time (no torn critical section).

The "blocks until release" tests should use a `threading.Event` to coordinate. Set timeout to 2 seconds.
