# T-6.3 — Transaction manager

## Goal
Wire `WriteLock` + `WAL` + (as-yet-implicit) page-writer into a `TransactionManager` that exposes `BEGIN` / `COMMIT` / `ROLLBACK`. The manager also tracks the active transaction (LSN range) and rejects nested BEGIN.

## Carry-forward NITs

- WAL frame codes RT_BEGIN / RT_COMMIT / RT_ROLLBACK exist; PAGE comes in T-6.4.
- WriteLock is single-writer reentrant; the manager holds it across BEGIN..COMMIT/ROLLBACK.
- Zero external deps.
- v0.1 is single-process — cross-process / cross-connection isolation comes much later (out of scope per DP-0).

## Files to Create / Modify

### `src/tinydb/tx/manager.py`

```python
"""Transaction manager — coordinates WAL writes + write lock.

Public surface:
    mgr = TransactionManager(pager, wal)
    mgr.begin()           # returns a TransactionContext (or raises)
    mgr.commit(tx)        # fsync WAL, mark COMMITTED, release lock
    mgr.rollback(tx)      # mark ROLLED BACK, release lock
    with mgr.transaction() as tx:
        # body runs inside the lock; auto-COMMIT on clean exit,
        # auto-ROLLBACK on exception.
        ...

The manager also exposes:
    mgr.active_tx -> TransactionContext | None
    mgr.recovery_pending -> bool   (set when an UNDO-only tx is found at startup)
"""
from __future__ import annotations
import contextlib
from dataclasses import dataclass

@dataclass
class TransactionContext:
    tx_id: int
    begin_lsn: int
    manager: "TransactionManager"

class NestedTransactionError(TinydbError):
    """Raised when BEGIN is called while another transaction is active.

    Per REQ-TRX-1 Scenario: nested transactions raise. v0.1 does NOT
    implement savepoints.
    """

class TransactionManager:
    def __init__(self, pager, wal: WAL) -> None:
        self._pager = pager
        self._wal = wal
        self._lock = WriteLock()
        self._next_tx_id = 1
        self._active_tx: Optional[TransactionContext] = None

    @property
    def active_tx(self) -> Optional[TransactionContext]: ...

    @contextlib.contextmanager
    def transaction(self):
        """Auto-COMMIT on clean exit; auto-ROLLBACK on exception."""
        tx = self.begin()
        try:
            yield tx
        except BaseException:
            self.rollback(tx)
            raise
        else:
            self.commit(tx)

    def begin(self) -> TransactionContext:
        """Acquire the write lock, append BEGIN record, return context.

        Raises NestedTransactionError if another tx is active.
        """
        if self._active_tx is not None:
            raise NestedTransactionError(
                "tinydb v0.1 does not support nested transactions"
            )
        with self._lock.acquire():
            tx_id = self._next_tx_id
            self._next_tx_id += 1
            begin_lsn = self._wal.append(RT_BEGIN, _encode_tx_begin(tx_id))
            self._active_tx = TransactionContext(tx_id, begin_lsn, self)
            return self._active_tx

    def commit(self, tx: TransactionContext) -> None:
        if self._active_tx is not tx:
            raise ValueError("commit(): not the active transaction")
        commit_lsn = self._wal.append(RT_COMMIT, _encode_tx_commit(tx.tx_id))
        self._wal.fsync()        # durability: COMMIT must reach disk
        self._active_tx = None
        # Lock release happens when `with self._lock.acquire():` exits
        # — the manager BEGIN holds it for the whole lifetime.

    def rollback(self, tx: TransactionContext) -> None:
        if self._active_tx is not tx:
            raise ValueError("rollback(): not the active transaction")
        rollback_lsn = self._wal.append(RT_ROLLBACK, _encode_tx_rollback(tx.tx_id))
        self._wal.fsync()
        self._active_tx = None
```

> Important: the WAL must fsync on COMMIT/ROLLBACK *before* the lock is released. Failure to fsync means a crash could lose the COMMIT decision → recovery treats it as aborted. Both branches fsync for safety.

## Tests Required

`tests/tx/test_manager.py` — at least 8 cases:

1. `mgr.transaction()` happy path → BEGIN + COMMIT records appear in WAL.
2. Exception inside `with mgr.transaction()` → BEGIN + ROLLBACK records appear; exception propagates.
3. Nested `with` raises `NestedTransactionError` after outer enters.
4. After commit, `mgr.active_tx is None` and lock is released (another thread can BEGIN).
5. After rollback, `mgr.active_tx is None` and lock is released.
6. `commit()` called with non-active tx raises `ValueError`.
7. `begin()` then no commit/rollback (caller forgot) → BEGIN is in WAL, lock leaked (documented behavior; gated by Python's GC eventually calling `__exit__` of the `with` for the lock).
8. Two threads: T1 holds BEGIN; T2's `mgr.transaction()` blocks; T1 commits → T2 proceeds.
9. `tx_id` increments monotonically across BEGIN/COMMIT cycles.
10. (bonus) Records carry the correct tx_id in their payload (decoded).
