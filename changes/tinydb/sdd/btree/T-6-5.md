# T-6.5 — Isolation (READ COMMITTED)

## Goal
Per REQ-TRX-4: tinydb provides READ COMMITTED isolation. In a single-process model with one write lock, this means:
- A SELECT running outside a transaction **sees only COMMITTED state** (committed data is durable; rolled-back data is gone).
- A SELECT running inside a transaction **also sees only COMMITTED state** (READ COMMITTED — no read-your-own-writes for v0.1).

Because tinydb v0.1 is single-process + single-writer (the write lock prevents concurrent writers), the only isolation guarantee we need to TEST is: uncommitted modifications in one transaction are invisible to a concurrent SELECT.

But "concurrent" in v0.1 means: Thread A holds the write lock and is partway through a transaction. Thread B cannot make a write — but it CAN read (Heap.scan does NOT require the write lock). Test: B's read doesn't see A's uncommitted inserts.

## Carry-forward NITs

- Single-process model: Thread B's SELECT must read committed heap state.
- Heap.scan() is read-only and does NOT acquire the write lock.
- T-6.6 (Recovery) is the one that hardens durability — T-6.5 just needs to verify *visibility* in-memory.

## Files to Create / Modify

### `src/tinydb/tx/manager.py` (extend — minimal)

For v0.1, SELECT queries don't acquire the write lock (B5 executor only acquires it during INSERT/UPDATE/DELETE). READ COMMITTED means we simply require the Heap pages be flushed to disk on COMMIT before any other reader can see them. Today, B5 already keeps heap pages in-memory until COMMIT — they're consistent because the writer holds the lock.

> Pragmatic decision: in single-process mode, READ COMMITTED is satisfied automatically because:
> 1. SELECT doesn't acquire the write lock.
> 2. The DML code path only modifies heap pages in-memory during a transaction.
> 3. Pages are flushed to disk on COMMIT.
> 4. Until COMMIT, the in-memory heap pages differ from disk; this in-memory state is visible only to threads that are PAST the write lock (which only the writer holds).
>
> Therefore a SELECT running concurrently will see the COMMITTED (on-disk) state.

So T-6.5 is **mostly test work** with minimal code changes. We may add a small helper `executor.execute_in_tx(plan, mgr)` that explicitly routes DML through the manager lock — but for SELECTs it stays lock-free.

### `tests/tx/test_isolation.py` (NEW)

## Tests Required

At least 8 cases:

1. **Reader sees committed** — Thread A commits an INSERT; Thread B starts after and sees it.
2. **Reader does NOT see uncommitted** — Thread A BEGIN + INSERT (no commit); Thread B starts SELECT → does NOT see A's row.
3. **Sequential read after commit** — A begins, inserts, commits; B reads → sees row.
4. **Sequential read after rollback** — A begins, inserts, rolls back; B reads → does NOT see row.
5. **Same-thread tx reads its own uncommitted** — A inside `mgr.transaction()` reads after its own insert → sees row (we DO support read-your-own-writes for the same connection — it's actually fine because the row is in memory).
6. **No dirty read across threads** — A is mid-tx (holds lock); B's SELECT bypasses the lock and reads; verify B does not see A's modifications.
7. **SELECT does not block on the write lock** — A holds BEGIN; B can run SELECT without blocking (verify with timeout 2s).
8. **Multiple concurrent BEGINs** — A holds BEGIN; B's `mgr.transaction()` is blocked until A finishes (write lock semantics).

Tests 1–7 verify READ COMMITTED. Test 8 verifies the write lock interaction.

### Helpers / Fixtures

Use threads + `threading.Event` to coordinate; set timeout 2s on Blocking-related waits (use `concurrent.futures.ThreadPoolExecutor` or `Event.wait(timeout=2)`).
