# T-5.3 — IndexScan + planner choice

## Goal
Replace `SeqScan` with `IndexScan` whenever the WHERE clause has a usable single-column equality or range predicate against a column covered by an index. Falls back to `SeqScan + Filter` when no index exists or the predicate shape doesn't match.

This is the batch where indexes first accelerate queries — measurable win on `WHERE id = N` with a unique index.

## Carry-forward NITs to incorporate (from T-5.2 + T-5.1)

1. `Executor` lives at `tinydb.executor.executor` (importable as `tinydb.executor.Executor`).
2. `Heap` is bound by reading `catalog._pager` and rebinding `Heap._head_pid` — keep this pattern (T-5.5 will formalize with `Heap.attach(...)`).
3. Project now uses parallel `items=(label_ast_node, ...)` array — preserve.
4. `Plan.table` is a `@property` on every leaf/wrapper; **IndexScan must also return the underlying table name** so the executor's heap/column-resolution code works.
5. Public `tinydb.executor` exports must remain: `Executor`, `plan`, `Plan`, `SeqScan`, `IndexScan`, `Filter`, `Project`, `Sort`, `Limit`, `Insert`, `Update`, `Delete`.

## New / Modified Files

### `src/tinydb/executor/index_scan.py` (NEW — implementation only)

```python
from __future__ import annotations
from typing import Iterator, Tuple, Any, Optional
from tinydb.storage.heap import Rid
from tinydb.storage.catalog import Catalog, TableMeta
from tinydb.types.codec import decode_value
from tinydb.index.manager import IndexManager
from tinydb.index.btree import BTreeIndex

class IndexLookup:
    """Resolve a column-rid lookup against an IndexManager.

    For a single-column equality (`col = literal`) uses ``BTreeIndex.search``;
    for a single-column range uses ``BTreeIndex.range``. Returns an
    iterator of ``(rid, key)`` pairs where ``key`` is the decoded scalar.
    For composite keys (T-5.5 wider ground) we may add `_range_tuple`
    later — not in this task.
    """

    def __init__(self, mgr: IndexManager, index: BTreeIndex, key_tag) -> None:
        self.mgr = mgr
        self.index = index
        self.key_tag = key_tag

    def equality(self, value: Any) -> Iterator[Tuple[Rid, Any]]:
        for rid in self.index.search(value):
            yield rid, value

    def range(self, lo: Any, hi: Any, *, lo_inclusive: bool = True,
              hi_inclusive: bool = True) -> Iterator[Tuple[Rid, Any]]:
        # Wrap BTreeIndex.range — it only supports lo_inclusive == hi_inclusive.
        # For the four boundary combinations, encode via `inclusive=True/False`
        # passing the tighter bound first when needed.
        if lo_inclusive and hi_inclusive:
            yield from ((rid, None) for rid in self.index.range(lo, hi))
        elif (not lo_inclusive) and (not hi_inclusive):
            # BTreeIndex exposes only (lo, hi, inclusive=True|False);
            # when inclusive=False means open interval (lo, hi).
            yield from ((rid, None) for rid in self.index.range(lo, hi, inclusive=False))
        elif lo_inclusive and not hi_inclusive:
            yield from ((rid, None) for rid in self.index.range(lo, hi, inclusive=False))
        else:  # (lo, hi]
            # Implement via `lo_exclusive` by descending past lo first.
            # For v0.1 we accept this as a known reduction: not all 4
            # boundary combos are wired; T-5.4 + tests cover the common
            # cases only.
            yield from ((rid, None) for rid in self.index.range(lo, hi))
```

> Pragmatic NIT: For T-5.3 only the two inclusive-flavor ranges need to work (`>= lo AND <= hi`, `> lo AND < hi`). Half-open inclusive-exclusive combinations are documented as "not supported" — log this in the report and add a test that asserts the right behavior per the brief's case list below.

### `src/tinydb/executor/index_plan.py` (NEW — planner predicate matcher)

```python
from __future__ import annotations
from dataclasses import dataclass
from typing import Optional, Tuple, Any
from tinydb.sql.ast import BinaryOp, ColumnRef, Literal, Expr, TypedLiteral, UnaryOp

@dataclass(frozen=True, slots=True)
class IndexablePredicate:
    """Result of extracting an indexable predicate from a WHERE clause.

    ``column`` is the name being matched; ``op`` is one of:
        '=', '<', '<=', '>', '>='
    ``value`` is the literal Python value being matched.
    For a range bound (e.g. `col BETWEEN a AND b`) this struct also
    carries ``hi`` / ``hi_op`` — only the columns-bound surface is
    supported here; multi-column OR-of-ranges is out of scope.
    """
    column: str
    op: str  # see above
    value: Any
    hi_op: Optional[str] = None
    hi_value: Optional[Any] = None

    Returns the IndexablePredicate or None. Composite conditions
    (AND of indexable ranges on same col → range; AND of two
    DIFFERENT columns → only the first surfaced, fall back to scan).
    """

```

**Algorithm** (T-5.3 spec):
- Walk `where` looking for `BinaryOp(col, op, literal)` or `BinaryOp(literal, op, col)`.
- If both operands are ColumnRef → not indexable (return None).
- Multiple top-level predicates joined by AND (`BinaryOp(AND, ...)`):
  - Same column on both sides → merge into a range (`lo <= col <= hi`).
  - Different columns → return None (planner falls back to SeqScan + Filter that evaluates both — the executor already handles this via the Filter op).
  - Range form `BETWEEN` and explicit AND both supported.
- `col IS NULL` and `col IS NOT NULL` are NOT indexable in v0.1 (carries forward from T-4.6 accepted deviation: NULL keys are skipped).

### `src/tinydb/executor/planner.py` (modify)
Replace the T-5.1 stub `_try_index_plan` with a real implementation:
- It accepts `(predicate: Expr, table_meta: TableMeta, catalog: Catalog)` and returns an `IndexScan` or None.
- Uses `IndexManager.get(table, [column])` to look up the index.
- If found AND the predicate is `extract_indexable` → `IndexScan(table=table_meta.name, index=idx_meta.name, lo=..., hi=..., lo_inclusive=..., hi_inclusive=...)`.
- Otherwise None.

For T-5.3 we don't have an `IndexManager` constructor wired in the executor. The simplest path: the planner takes an optional `IndexManager` param on `plan()`. Default to None; if provided, use it. Provide an `Indexer.indexer` attribute on the executor to cache:
```python
def plan(stmt, catalog, indexer: Optional[IndexManager] = None) -> Plan: ...
```

In `Executor.__init__` accept `indexer: Optional[IndexManager] = None`.

## Tests Required (RED → GREEN)

`tests/executor/test_index_plan.py` — at least 10 cases:

1. `extract_indexable` parses `col = 1` → IndexablePredicate(col, '=', 1)
2. Parses `1 = col` (literal on left)
3. Parses `col >= 5` → op='>='
4. Parses `col < 100` → op='<'
5. AND of same column → range: `col >= 5 AND col <= 10`
6. AND of different columns → None
7. `col IS NULL` → None (not indexable)
8. `col BETWEEN 5 AND 10` → range
9. Arithmetic on column (`col + 1 = 5`) → None
10. Function call (`lower(col) = 'x'`) → None

`tests/executor/test_index_scan.py` — at least 8 cases (integration tests building real indexes, running `Executor.execute(plan(sql))`):

1. Single-column unique index, `WHERE id = 7` → uses IndexScan (assert via `IndexScan` in plan).
2. Same query WITHOUT an index → SeqScan (no `IndexScan` in plan tree).
3. `WHERE age >= 18` with non-unique index → IndexScan with lo=18, lo_inclusive=True.
4. `WHERE age BETWEEN 18 AND 30` → range scan.
5. Insert/update/delete (T-5.5 will properly wire DML; for T-5.3 we use `IndexManager.on_insert` directly in fixtures to populate the index):
   - Build index on `users(name)`
   - Insert (Rid, row) via heap.insert + index_mgr.on_insert
   - Run `SELECT * FROM users WHERE name = 'alice'` → returns the right row.
6. After deletion (`on_delete`), a search returns `[]`.
7. After update (`on_update` with same key), the old key now returns `[]` and the new key returns the row.
8. NULL key in indexed column → planner falls back to SeqScan (NULL keys skipped by index manager → not in the B-tree).
9. Equality on a column not in any index → falls back.
10. Heap has NO rows AND index is empty → query returns `[]` not crash.
