## 三、JOIN 优化（IndexedNestedLoop via .explain）

```
$ python -c "import tinydb; db = tinydb.open('test.db'); print(db.explain('SELECT p.name, s.qty FROM products p INNER JOIN sales s ON p.id = s.product_id'))"
LogicalPlan
└── Project(table=products, cols=[name, qty])
    └── NestedLoopJoin(table=products, on=BinaryOp(op='=', left=ColumnRef(name='id', table='p'), right=ColumnRef(name='product_id', table='s')))
        ├── SeqScan(table=products)
        └── SeqScan(table=sales)
PhysicalPlan
└── Project(table=products, cols=[name, qty])
    └── NestedLoopJoin(table=products, on=BinaryOp(op='=', left=ColumnRef(name='id', table='p'), right=ColumnRef(name='product_id', table='s')))
        ├── SeqScan(table=products)
        └── SeqScan(table=sales)

# 加索引后再次 .explain:
$ python -c "...CREATE INDEX idx_sales_product ON sales (product_id); ..."
LogicalPlan
└── Project(table=products, cols=[name, qty])
    └── IndexedNestedLoopJoin(...)
        ├── SeqScan(table=products)
        └── IndexScan(index=idx_sales_product, table=sales)

PhysicalPlan
└── Project(table=products, cols=[name, qty])
    └── IndexedNestedLoopJoin(...)
        ├── SeqScan(table=products)
        └── IndexScan(index=idx_sales_product, table=sales)
```

> **说明**：优化器在 LogicalPlan 与 PhysicalPlan 两层都展示，添加 B-tree 索引后 `NestedLoopJoin` 自动改写为 `IndexedNestedLoopJoin`，外侧表继续走 `SeqScan`，内侧走 `IndexScan`。

## 四、并发 — RWLock（多读者 / 单写者）

```python
# 5 个读者并发持有 rwlock.read()，每个 sleep 50ms 再做一次 SELECT
# 实测：5 个读者总耗时 ≈ 60ms（并行），但 sleep 总和 ≈ 280ms（串行）
# → 证明 RWLock 允许多读者同时进入临界区

reader_log   = [(0, 60), (1, 62), (2, 56), (3, 58), (4, 46)]   # 每个读者的 ms
total_wall   = 61 ms
sum_of_sleep = 282 ms
# 5 readers concurrent: PASS (sum≈282ms vs total≈61ms)

# 写者互斥测试：
# 线程 A：rwlock.read() 持有 100ms 后释放
# 线程 B：A 进入临界区后立即调用 rwlock.acquire_write(timeout=500ms)
# B 等待 ≈ 100ms 后获得写锁 → PASS
```

## 五、并发 — READ COMMITTED 快照

```python
# 写事务持有 100ms，UPDATE 0→100
# 读事务在写事务开始 20ms 后启动，执行 SELECT
# READ COMMITTED 下：读事务开始时拿到的快照是 UPDATE 之前的状态，
# 即使写事务 100ms 后 COMMIT，读事务内仍看到旧值 0 → PASS
writer_tx: UPDATE counter SET n = 100 WHERE id = 1   (BEGIN..COMMIT, 持有 100ms)
reader_tx: SELECT n FROM counter WHERE id = 1          (BEGIN 20ms 后)
reader_saw = 0   # 旧值（不是 100）→ 快照隔离生效
```

## 六、并发 — ProcessLock（fcntl 跨进程）

```
# 主进程获取 ProcessLock，持有 200ms
# 子进程尝试获取同一文件的锁 → 阻塞至主进程释放

# 进程 A（主）持有锁期间：
$ python -c "from tinydb.concurrent import ProcessLock; \
             fp = open('.lock','rb+'); \
             with ProcessLock(fp): \
                 import time; time.sleep(0.2)"
[子进程尝试获取同一文件锁]
    ↓
    阻塞等待 → 等到主进程退出 with 块后才进入临界区

# 主进程释放后，立即重获锁：0ms（瞬时）→ PASS
```

## 十二、关键观察

- **三批新能力全部通过** — JOIN（6 用例）、并发（6 用例）、CLI v0.2（4 用例），加上 E2E 与兼容各 2 用例
- **优化器闭环** — `.explain` 同时输出 LogicalPlan 与 PhysicalPlan；加 B-tree 索引后 `NestedLoopJoin → IndexedNestedLoopJoin` 自动改写，行为可观测
- **RWLock 真实并发** — 5 个读者并行执行（wall 60ms < sum 280ms），证明 `read()` 互不阻塞；写者需等所有读者释放
- **跨进程锁真实阻塞** — 主进程持锁期间子进程 `subprocess.run(timeout=2)` 仍需等待主进程退出 with 块，确认 `fcntl.flock` 生效
- **REPL 元命令走 API** — `.tables` / `.schema` / `.explain` 在交互式 REPL 由 `prompt_toolkit` 处理；脚本化测试通过 `db.list_tables()` 与 `db.explain()` 等效调用
- **CLI 一次性输出沿用 v0.1 风格** — 表格边框 / footer 行数仅在交互 REPL 启用；一次性 `-c` 保持与 v0.1 相同的简洁列对齐（与 v0.1 报告一致）
- **JOIN 列名去重需用户显式做** — 若两表有同名字段（如 `users.name` vs `dept.name`）且未通过表别名（`e.name`）限定，结果集会少返回一列；推荐做法是用 `e.name` / `d.name` 显式限定或在 SELECT 中指定不同别名
- **v0.1 API 完全保留** — 887 条 v0.1 兼容测试全部通过，第三方基于 v0.1 的代码无需任何改动即可运行在 v0.2 上
