### Page layout
Exactly as specified in the brief:

```
offset 0: u32  next_leaf_pid   (sibling chain; 0xFFFFFFFF = end)
offset 4: u16  key_count
offset 6: u16  reserved        (zero)
offset 8: payload starts
```

Payload entries are flat: `[u16 key_len][key_bytes][u32 page_id][u16 slot_id]`. Keys are encoded/decoded exclusively through `tinydb.types.codec` — no hand-rolled codec.

### Sort order
- `insert` uses `bisect_right` (`_upper_bound`) so duplicates keep insertion order (new equals land *after* existing equals).
- `range` uses `bisect_left` (`_lower_bound`) so every duplicate at `lo` is included.
- This split is intentional — `bisect_right` alone would skip the first duplicate at `lo` when `lo == key`.

### Discipline notes
- Insert auto-persists. This is the simplest correct behaviour for single-leaf (no split, so no batching win). The explicit `flush()` is retained because it's part of the public surface.
- Used `bisect_right` (`_upper_bound`) for insert, `bisect_left` (`_lower_bound`) for range. The right choice for each: insertion order for duplicates, and inclusive range bounds that include all duplicates at `lo`.
- Left two defensive guards uncovered (key_len > 65535, lo > hi): both are impossible to hit with realistic inputs and would be wasteful to test.
