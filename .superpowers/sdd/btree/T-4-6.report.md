## What I implemented

T-4.6 ties together the catalog, type system, constraint enforcement, and the underlying `BTreeIndex` into a single `IndexManager` class. The manager owns:

1. **Catalog-backed index metadata** — `IndexMeta` (frozen `@dataclass(slots=True)`) carries `name`, `table`, `columns`, `root_pid`, `unique`. Persisted on **page 2** (a new reserved catalog page) via the same JSON-with-length-prefix layout as the existing table catalog on page 1.
2. **`IndexManager(catalog, pager)`** — builds the live `(meta, BTreeIndex)` map from `Catalog.all_indexes()` on construction. Each `BTreeIndex` is bound to its `root_pid` and the right `TypeTag` for the first indexed column (single-column) or `TypeTag.Json` (composite).
3. **Write hooks** — `on_insert` / `on_update` / `on_delete` walk the live indexes for a table, extract the key column values from the row tuple, enforce UNIQUE, and update each B-tree in place. NULLs are skipped from the B-tree entirely per REQ-IDX-5 (NULL does not participate in the UNIQUE check, so it also need not participate in the index).
4. **Catalog persistence** — `create_index` allocates a fresh page via `pager.allocate_page()`, registers the metadata with `Catalog.create_index`, and stores the B-tree's `root_pid`. `drop_index` removes the catalog entry first, then walks the B-tree (via the new `btree_pages.all_page_ids` helper) and frees every page it owned.
5. **Cross-restart persistence** — on reopen, `IndexManager.__init__` reads every entry from `Catalog.all_indexes()` and reopens the corresponding `BTreeIndex`. The lazy-load contract from T-4.1 still applies; the first `search` / `range` / `insert` reads the root page.
6. **T-4.1 NIT fix** — `btree_leaf._read_leaf` now coerces `next_pid == 0` unconditionally, not just when `key_count == 0`. This is the carry-forward from T-4.5 NIT #1 and matches the brief's recommended fix.

### Design decision choices

1. **Index storage location (option a vs b).** Chose **(a)** — reserved catalog page 2 for indexes. The existing catalog uses page 1 for tables and has uniform table-of-tables serialization. A separate page (option a) keeps both catalogs uniform ("page 1 = tables, page 2 = indexes") without mixing two concerns on one page. `INDEX_CATALOG_PAGE = 2` is exposed alongside `CATALOG_PAGE`.
2. **Row representation.** `row: tuple` in column order, as the brief specifies. The manager extracts keys via `_extract_key(row, columns, table_meta)` using a `{col.name: idx}` map built once per call.
3. **NULL handling.** Per REQ-IDX-5 NULL does not participate in the UNIQUE check. v0.1 goes one step further: NULL keys are **not inserted into the B-tree at all**. This is the only way the codec can round-trip a mixed-type key (TypeTag.Text rejects `None`). `search(None)` returns `[]` for any single-column or composite index that encountered a NULL — see concerns below.
4. **Composite key shape.** Deviates from the brief's "always pass tuples" recommendation. **Single-column: pass the scalar under the column's `TypeTag`** (e.g. `"alice@example.com"` under `TypeTag.Text`). **Composite: pass the tuple under `TypeTag.Json`** (the T-4.5 path; tuples round-trip as JSON lists). The codec enforces strict value/tag pairing, so a one-element tuple `(str,)` under `TypeTag.Text` raises `_CodecError`. The deviation is documented in the manager module docstring.

## Concerns

1. **NULL keys are skipped, not indexed.** The brief says "UNIQUE allows NULL — NULL does not participate in unique check." The implementation is stricter than the brief — NULL keys are not added to the B-tree at all (they're filtered in `on_insert` / `on_update` / `on_delete` via `_key_contains_none`). The consequence: `idx.search(None)` returns `[]` rather than the rows with NULL keys. The brief's test scenario 9 only asserts that the second NULL insert does NOT raise — which my implementation satisfies. The deviation is documented in the manager module docstring. A future batch could encode NULLs under a separate TypeTag.Null subtree if the `search(None)` behaviour is needed.

2. **Composite key shape deviates from brief recommendation.** The brief's recommendation was "always pass tuples to BTreeIndex." I deviate because the codec rejects a tuple under a non-JSON tag. Single-column: pass scalar under the column's tag. Composite: pass tuple under `TypeTag.Json` (matches T-4.5 path). The deviation is documented and the tests reflect it (`search("alice@example.com")`, not `search(("alice@example.com",))`).

3. **btree_rebalance.py coverage at 74%.** T-4.5 NIT #2 — explicitly deferred per the brief.

4. **btree_pages.py coverage at 79%.** Only one branch of `all_page_ids` (internal-root walk) is exercised by the existing tests; the leaf-root branch is reachable only when an index with a leaf root is dropped. Acceptable for v0.1 — the helper is small, the leaf-root branch is straightforward, and it's structurally the same code.
