## Goal

Create `src/tinydb/index/manager.py` with `IndexManager(catalog)` and the **per-write hook contract**:

1. **`create_index(name, table, columns, *, unique=False) -> None`** — allocate a new B-tree on a fresh `pager.allocate_page()`, register it in the catalog (so it survives restart), and start maintaining it on every `INSERT/UPDATE/DELETE` for that table.
2. **`drop_index(name) -> None`** — remove from catalog and free the B-tree pages.
3. **`get(table, columns) -> BTreeIndex | None`** — look up an existing index by `(table, columns)`.
4. **`list_indexes(table=None) -> list[str]`** — list all index names, optionally filtered by table.
5. **UNIQUE enforcement** — on `INSERT/UPDATE`, look up any unique index for the affected columns; if the key already exists, raise `ConstraintViolation`.
6. **Persistence across restart** — index metadata (root_pid, key columns, unique flag) is persisted via the catalog; on reopen, `IndexManager.__init__` rebuilds the `BTreeIndex` objects from disk.
7. **Write hook contract** — `IndexManager.on_insert(table, rid, row)`, `on_update(table, rid, old_row, new_row)`, `on_delete(table, rid, row)`. The executor (Batch 5) will call these hooks on every write.

## Public interfaces

```python
# tinydb.index.manager
class IndexManager:
    def __init__(self, catalog: Catalog, pager: Pager) -> None: ...
    def create_index(self, name: str, table: str, columns: tuple[str, ...], *, unique: bool = False) -> None: ...
    def drop_index(self, name: str) -> None: ...
    def get(self, table: str, columns: tuple[str, ...]) -> BTreeIndex | None: ...
    def get_by_name(self, name: str) -> BTreeIndex | None: ...
    def list_indexes(self, table: str | None = None) -> list[str]: ...
    def on_insert(self, table: str, rid: Rid, row: tuple) -> None: ...      # row is encoded column values
    def on_update(self, table: str, rid: Rid, old_row: tuple, new_row: tuple) -> None: ...
    def on_delete(self, table: str, rid: Rid, row: tuple) -> None: ...
```

## Catalog persistence shape

Add an index-metadata table to the catalog. The catalog currently has tables; add indexes:

```python
@dataclass(frozen=True)
class IndexMeta:
    name: str
    table: str
    columns: tuple[str, ...]
    root_pid: int
    unique: bool
```

Store in catalog page 2 (page 1 currently holds table metadata; reserve page 3 for indexes; page 4+ free). On `Catalog.create_index`, serialize the `IndexMeta` and write to the indexes page. On `Catalog._load_from_disk`, deserialize.

Alternative (simpler): use a separate "system table" approach where the catalog treats indexes as a meta-table. Either works — pick whichever fits the existing catalog code more naturally.

## Design decisions the implementer must make

1. **Index metadata storage location in catalog.** Two options:
   - (a) Reserve catalog pages 3 for indexes; serialize as a flat list.
   - (b) Treat indexes as a "system table" with name `_tinydb_indexes`; store alongside user tables.
   - **Recommendation:** (b) is more uniform with the existing catalog code; if the catalog doesn't already use a uniform table-of-tables shape, fall back to (a).
2. **Row representation.** The brief uses `row: tuple` — encoded column values matching the table's column order. `on_insert(table, rid, row)` receives the row from the executor; `IndexManager` extracts the relevant keys via column indices. Document this contract.
3. **NULL handling in UNIQUE.** Per REQ-IDX-5: "UNIQUE 允许 NULL — NULL 不参与唯一性检查". Implementation: before insert, if the indexed value is `None`, skip the duplicate check (NULL is not a duplicate of NULL for unique purposes).
4. **Composite key extraction.** When `columns = ("a", "b")`, the B-tree key is `tuple(row[col_idx_a], row[col_idx_b])`. Single-column: `tuple(row[col_idx])` or just `row[col_idx]`? **Recommendation:** always pass tuples to `BTreeIndex` (single-element tuple is unambiguous; matches the T-4.5 composite-key path).
