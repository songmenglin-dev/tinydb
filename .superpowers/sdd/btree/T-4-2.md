# Task Brief — T-4.2: B-tree node split + internal nodes

## Where this fits

Batch 4 (B-tree Index) of the tinydb change. T-4.2 is the second of six tasks.
Predecessor: T-4.1 (single-leaf B-tree, `7e6b975`).
Successors: T-4.3 search/range via tree walk, T-4.4 delete + rebalance, T-4.5 composite keys, T-4.6 IndexManager.

## Goal

Extend `src/tinydb/index/btree.py` so the B-tree grows beyond a single leaf:

1. **Internal node type** — `(key, child_pid)` entries; child pages can themselves be leaves or internals.
2. **Split on insert overflow** — when a leaf would exceed the page-size capacity, split into two leaves, promote the middle key, and link via `next_leaf_pid` chain.
3. **Recursive root split** — if the root is the leaf that splits, create a new internal root. If an internal node splits, push the middle key up.
4. **Persistence of all node kinds** — every node, leaf or internal, lives in a `Pager` page and survives `close()`/`open()`.

## Must-fix NITs from T-4.1 review (carried forward)

The T-4.1 reviewer flagged two forward-looking NITs that **must be addressed in T-4.2**:

1. **Tail-leaf sentinel (`btree.py:110`).** A freshly allocated leaf page decodes `next_leaf_pid == 0` from zero-filled bytes, and `_write_leaf` then writes `0` to disk — not `NO_NEXT`. Today harmless (T-4.1 single leaf, pointer never dereferenced). After T-4.2 introduces siblings, the sibling chain must use `NO_NEXT` (`0xFFFFFFFF`) as the tail marker.
   - **Fix in T-4.2:** in `BTreeIndex.__init__` (or a new `_init_new_leaf` helper), when constructing a fresh leaf for the first time, ensure `leaf.next_leaf_pid = NO_NEXT` and persist explicitly with the sentinel. Also, on `_ensure_loaded`, if `next_leaf_pid == 0` AND the page is otherwise empty (`key_count == 0`), treat it as a fresh-tail leaf and coerce to `NO_NEXT` in memory (defensive against the existing on-disk bug for indexes created in T-4.1).
   - Add a regression test: insert one key, flush, reopen, assert `BTreeIndex._leaf.next_leaf_pid == NO_NEXT`.

2. **Capacity guard before split (`btree.py:124`).** The T-4.1 brief's "64 entries per leaf page fits under 4 KB" claim is wrong: 64 entries × ~77 bytes (64-char Text key) ≈ 4936 bytes > 4096 page. T-4.2 must add an **explicit capacity check** in `_write_leaf` that raises a structured `BTreeOverflowError` (subclass `TinydbError`, new) when the entry would overflow the page. The split algorithm must catch this overflow signal and trigger a split.
   - Define `BTreeOverflowError(TinydbError)` in `src/tinydb/index/btree.py` (small, focused).
   - Add a regression test: 64-char Text key with ~50 entries → expect `BTreeOverflowError` from raw `_write_leaf`; with the new split path enabled, the same workload creates internal nodes and `range()` returns all entries.

Optional (logged NITs, may fix in T-4.2 or defer):

3. **`_read_leaf(pid, key_type)` deletes its `key_type` arg (`btree.py:83`).** Caller-controlled today. Either remove the misleading parameter from the signature, or actually validate the on-wire tag against the supplied one. **Recommendation:** remove the parameter entirely (no caller relies on validation, and the on-wire tag is authoritative).

4. **`assert` for internal invariants (`btree.py:178`, `_lower_bound`, `_upper_bound`).** Promote to `RuntimeError` so `python -O` doesn't silently strip the guards.

Internal additions:

```python
@dataclass
class InternalNode:
    keys: list[Any]       # separator keys; n children → n-1 keys
    children: list[int]   # child page ids (parallel to keys: keys[i] separates children[i] and children[i+1])

# Module-level:
def _read_internal(pager: Pager, pid: int) -> InternalNode: ...
def _write_internal(pager: Pager, pid: int, node: InternalNode) -> None: ...

# Within BTreeIndex (private):
def _split_leaf_if_needed(self) -> None: ...    # called from insert()
def _split_internal_if_needed(self, pid: int) -> None: ...
def _find_child(self, key: Any, internal: InternalNode) -> int: ...   # index into children[]
```

New exception:

```python
class BTreeOverflowError(TinydbError):
    """Raised by _write_leaf / _write_internal when an entry would not
    fit in the remaining bytes of a page.  The split algorithm catches
    this and re-tries after redistributing the data across two pages.
    """
```

## Page layout for an internal node

```
offset 0: u16  key_count       (number of separator keys = children - 1)
offset 2: u16  reserved        (zero)
offset 4: u32  first_child_pid (children[0])
offset 8: payload entries:
    [u16 key_len][key_bytes][u32 child_pid] ...   (repeated key_count times)
    where child_pid[i] sits between keys[i-1] and keys[i]
    (so keys has key_count entries; children has key_count+1 entries)
```

Payload entries are flat: `[u16 key_len][key_bytes][u32 child_pid]`. Keys encoded via the same `tinydb.types.codec.encode_value` / `decode_value` used for leaves.

## Split algorithm (insert path)

```
insert(key, rid):
    _ensure_loaded()
    _insert_into_subtree(self._root_pid, key, rid)
    _persist()  # persist the (possibly new) root

_insert_into_subtree(pid, key, rid):
    node = _read_node(pid)
    if node is LeafNode:
        pos = bisect_right(node.keys, key)
        node.keys.insert(pos, key)
        node.rids.insert(pos, rid)
        try:
            _write_leaf(pager, pid, node, key_type)
        except BTreeOverflowError:
            _split_leaf(pid)   # may raise if even mid-leaf overflows (e.g. one entry > page)
    else (InternalNode):
        child_idx = _find_child(key, node)
        _insert_into_subtree(node.children[child_idx], key, rid)
        # After return, the child may have split; check by re-reading
        # if children[child_idx] now has a sibling link and re-balance.
        # For T-4.2, accept BTreeOverflowError from _write_internal and split.

_split_leaf(pid):
    leaf = _read_leaf(pager, pid, key_type)
    mid = len(leaf.keys) // 2
    left = LeafNode(keys=leaf.keys[:mid], rids=leaf.rids[:mid], next_leaf_pid=NO_NEXT)
    right = LeafNode(keys=leaf.keys[mid:], rids=leaf.rids[mid:], next_leaf_pid=leaf.next_leaf_pid)
    right_pid = pager.allocate_page()
    _write_leaf(pager, pid, left, key_type)
    _write_leaf(pager, right_pid, right, key_type)
    # If pid is root, create new internal root. Else push key into parent.
    if pid == self._root_pid:
        new_root_pid = pager.allocate_page()
        sep_key = right.keys[0]
        new_root = InternalNode(keys=[sep_key], children=[pid, right_pid])
        _write_internal(pager, new_root_pid, new_root)
        self._root_pid = new_root_pid
    else:
        # Caller (parent's _insert_into_subtree) will push the separator
        # via a follow-up _write_internal or by direct edit.
        raise _SplitSignal(mid_key=right.keys[0], new_pid=right_pid, old_pid=pid)
```

The exact propagation shape can be refined during implementation — what matters is:

- After the work, `range()` over a tree with >64 entries still returns every rid in order.
- After close/reopen, every rid is still there.
