### Page layouts

**Leaf** (12-byte header):

```
offset  0: u8  node_type              (= 0x01)
offset  1: u24 reserved               (zero)
offset  4: u32 next_leaf_pid          (sibling chain; 0xFFFFFFFF = end)
offset  8: u16 key_count
offset 10: u16 reserved               (zero)
offset 12: payload start
    entries: [u16 key_len][key_bytes][u32 page_id][u16 slot_id]
```

**Internal** (12-byte header):

```
offset  0: u8  node_type              (= 0x02)
offset  1: u24 reserved               (zero)
offset  4: u16 key_count              (= len(children) - 1)
offset  6: u16 reserved               (zero)
offset  8: u32 first_child_pid        (children[0])
offset 12: payload start
    entries: [u16 key_len][key_bytes][u32 child_pid]
```

The brief's locked leaf layout started `next_leaf_pid` at offset 0; I
added a 1-byte type tag at offset 0 and shifted `next_leaf_pid` to
offset 4 so the reader can dispatch on the type byte.  The internal
layout puts the type byte at offset 0 and shifts everything by one
byte relative to the brief's spec (key_count at offset 4 instead of
offset 0).  This deviation is documented because the brief's two
layouts overlap in their use of bytes 0..4 and there is no other way
to distinguish a leaf from an internal on disk.  See "Implementation
note: node-type tag" below.

### Split algorithm

`BTreeIndex.insert(key, rid)`:
1. `_ensure_loaded()` — lazy-load root (leaf or internal).
2. `_insert_into(root_pid, root_view, key, rid)` — recursive descent.
   * **Leaf**: insert via `_upper_bound`, then try `_write_leaf`.  If
     `BTreeOverflowError` is caught, call `_split_leaf` which writes
     the two halves and returns `(right.keys[0], right_pid)`.
   * **Internal**: descend into the child whose subtree owns `key`.  If
     the recursive call returns a `(push_up_key, right_pid)` tuple,
     splice the new separator into this internal node, then try
     `_write_internal`.  Catch `BTreeOverflowError` and split with
     `_split_internal` (middle key pushed up, NOT included in either
     side).
3. If the root split returned a `(push_up_key, right_pid)`, allocate a
   fresh internal-node root with `[push_up_key]` and `[old_root,
   right_pid]` as its children, and rewrite `self._root_pid`.

`_split_leaf` also rewrites the left page's `next_leaf_pid` to point
at the newly-allocated right page so the sibling chain stays walkable
across the split boundary.  The right page inherits whatever sibling
the original leaf had.

### `range()` for T-4.2

The brief said "T-4.2 keeps range single-leaf if split creates sibling
leaves not yet linked".  This task DOES link the sibling chain, so
`range` was extended to walk it: it descends to the leftmost leaf
whose key might satisfy `lo`, then walks `next_leaf_pid` until keys
cross `hi`.  T-4.3 will replace this with a true tree-walking range
scan.

## Implementation note: node-type tag

The brief's locked leaf layout (`next_leaf_pid` at offset 0) and
internal layout (`key_count` at offset 0) overlap in their use of
bytes 0..4; without a discriminator a freshly-allocated all-zero page
looks like both an empty leaf AND an empty internal.  I added a
1-byte node-type tag at offset 0 (`0x01` for leaf, `0x02` for internal)
and shifted every other field by one byte.  This is a documented
deviation from the brief's literal layout but matches its intent
(distinct on-disk formats that can be read back unambiguously).

Backward compatibility: `_read_leaf` and `_ensure_loaded` accept
`0x00` (all zeros) as "freshly allocated, never written" so the
constructor remains zero-I/O for the lazy-load contract.  Once the
first insert happens, `_write_leaf` writes the LEAF type byte and the
in-memory view becomes authoritative.

## Implementation note: `BTreeOverflowError` location

The brief said "Define `BTreeOverflowError(TinydbError)` in
`src/tinydb/index/btree.py`".  Putting it there created a circular
import — `btree_leaf.py` and `btree_internal.py` need it (they raise
it), but they must not import from `btree.py` (which imports them).
I moved the class to `src/tinydb/errors.py` and re-export it from
`src/tinydb/index/__init__.py` and `src/tinydb/index/btree.py` so the
public surface (`from tinydb.index import BTreeOverflowError`) is
unchanged.  The brief noted this move as "optional" ("added to
`tinydb.errors.__all__` if you want it visible there").
