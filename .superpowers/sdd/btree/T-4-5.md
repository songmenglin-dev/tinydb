## Goal

Extend `BTreeIndex` to accept **tuple keys** (e.g. `(last_name, first_name)`) and implement:

1. **Lexicographic comparison** for tuples — `(a, b) < (a, c)` is decided by `b < c`; `(a, b) < (a1, ...)` by `a < a1`.
2. **Prefix queries** — `range(lo, hi, inclusive=True)` where `lo` is a single value and the index key is `(last_name, first_name)` returns every entry whose `last_name == lo`, regardless of `first_name`.
3. **Multi-column range** — `range(lo_tuple, hi_tuple)` returns entries with full key in `[lo_tuple, hi_tuple]`.

The key encoding layer (`tinydb.types.codec.encode_value`) already serializes tuples if the column tag is composite. **T-4.5 focuses on the comparison + search semantics**, assuming the keys arrive at the index as Python tuples.

## Comparison semantics

The B-tree's existing comparison is `key_a < key_b` via Python's natural `<`. For tuples:

- `(1, 2) < (1, 3)` → True (compare position 1)
- `(1, 2) < (2, 1)` → True (compare position 0)
- `(1, 2) < (1, 2)` → False (equal)
- `(1,) < (1, 2)` → True (shorter tuple, prefix match)
- `(1, 2) < (1,)` → False (longer is greater)

Python's tuple comparison is **already lexicographic**. So `_lower_bound` / `_upper_bound` Just Work without changes. T-4.5 verifies this with tests.

## Prefix-search semantics for `range`

When the B-tree index key is `(last_name, first_name)`:
- `range("Smith", "Smith")` (inclusive=True) — returns every entry whose `last_name == "Smith"`. This works because tuple comparison `(last_name, first_name) < ("Smith", anything)` is False iff `last_name < "Smith"` — so `_lower_bound(("Smith",))` lands on the first entry with `last_name == "Smith"`, and `range` continues until `last_name > "Smith"`.
- `range(("Smith", "A"), ("Smith", "M"))` — returns entries with `last_name == "Smith"` AND `"A" <= first_name <= "M"`.

This is exactly how multi-column indexes work in real DBs. The semantics fall out naturally from tuple comparison; **no code change to `range` is needed** — only the test scenarios must exercise it.
